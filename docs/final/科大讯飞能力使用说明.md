# 科大讯飞能力使用说明

**适用于:** 比赛答辩 / PPT 附件 / 评委查阅
**版本:** 比赛版 v3 (方案 A+)
**日期:** 2026-05-24

---

## 概述

本项目（智能学习助手）已接入科大讯飞 AI 能力，并以国产模型可配置、多模态可扩展的方式展示。以下说明系统已实现的能力、配置方法及未来扩展方向。

---

## 1. 已实现：星火大模型接入

### 接入方式

本项目通过科大讯飞星火大模型 HTTP 接口实现接入，完全兼容 OpenAI SDK：

```
Base URL: https://spark-api-open.xf-yun.com/v1
接口:     /v1/chat/completions
鉴权:     APIPassword
模型:     generalv3.5（可配置为 generalv4.0 等）
```

### 配置方法

在 `.env` 文件中配置以下参数即可切换至星火模型：

```env
LLM_PROVIDER=spark
SPARK_ENABLED=true
SPARK_API_PASSWORD=<您的APIPassword>
SPARK_MODEL=generalv3.5
```

或通过前端"系统设置"页面，选择"星火 Spark"并填入 APIPassword 后保存。

### 当前默认模型

为保证比赛演示的端到端稳定，**当前默认使用 DeepSeek（deepseek-chat）**。
切换至星火模型仅需修改一条配置项，所有代码和测试均已就绪。

---

## 2. 模型层三层 Fallback

系统设计了完善的三层容灾机制：

```
第一层: Spark (星火)    ← 国产模型，优先尝试
第二层: DeepSeek        ← 默认稳定模型
第三层: Mock             ← 离线容灾，确保系统永远可用
```

当某一层不可用（密钥缺失、网络超时、服务异常）时，系统自动降级到下一层，
并在响应中记录 fallback 来源和原因，确保学习流程不中断。

## 4. 微课讲解稿（可选轻量展示）

系统可选支持"生成微课讲解稿"功能：

- **输入**: 当前 AI 回答 + 引用来源 + 学习路径
- **输出**: 60 秒课堂讲解稿文本
- **展示**: "数字人课堂助手"卡片内
- **依赖**: 无外部 API，纯文本生成
- **后续**: 可接入 TTS 将讲解稿转为语音，或接入数字人进行口播

> 注意：当前版本**不**调用科大讯飞 TTS 或数字人视频 API，
> 讲解稿为纯文本输出。语音合成和数字人属于明确的下一步扩展方向。

## 5. 答辩推荐表述

### 针对评委可能的提问

**Q: 你们是否使用了国产大模型？**

A: 是的。我们已完整接入科大讯飞星火大模型，通过 OpenAI-compatible HTTP 接口实现。
系统支持一键切换至星火模型，只需配置 APIPassword。
当前演示为保障稳定性采用 DeepSeek，但所有接入代码、配置和
测试均已就绪。系统还设计了 Spark → DeepSeek → Mock 三层 fallback 机制。

**Q: 你们有多模态或数字人能力吗？**

A: 我们采用轻量化策略。系统预留了数字人课堂助手入口，
并可基于当前回答生成微课讲解稿——这是语音合成和数字人的前置内容。
语音合成和数字人接口配置已在系统中预置，
后续接入科大讯飞相关服务即可将讲解稿转为语音或数字人微课。

**Q: 讯飞能力在你们项目中的定位是什么？**

A: 分三层：
- **已交付**: 星火大模型可配置接入，国产模型支撑
- **当前可选**: 微课讲解稿，为语音/数字人提供前置内容
- **未来扩展**: TTS 语音合成、OCR 教材扫描、完整数字人助手

我们不夸大当前实现，也不回避技术储备。

## 6. 重要说明

1. **不夸大**: 当前内容并非全部由讯飞生成。主流回答由 DeepSeek 生成，
   SparkProvider 作为可切换选项。

2. **不泄露密钥**: 所有 API Key、APIPassword、JWT Secret
   均不包含在代码仓库中，仅在 `.env` 文件中本地配置。

3. **演示优先**: 当前比赛版以学习闭环稳定性为首要目标，
   因此默认使用已验证的 DeepSeek 模型。

4. **技术诚实**: 星火接入代码完整可用，但未使用真实密钥进行端到端验证。
   答辩时应说明代码路径就绪，而非声称已完成生产级验证。
